三种架构方案对比

选择一个作为 Agent 决策引擎的顶层架构

A

极简单循环 — 一个 AgentLoop 类驱动一切

所有逻辑集中在一个类中：状态机、工具字典、消息列表、循环逻辑。整个引擎只有一个核心文件。

优点

  • 极简，~300 行核心代码
  • 理解成本最低
  • 调试直观

缺点

  • 难以扩展
  • 各部分耦合
  • 无法独立测试子模块
B

模块化组件 — 每个关注点独立为类（推荐）

DecisionLoop / StateManager / ToolRegistry / MemoryManager / Reflector / Observer 各司其职，通过明确定义的接口协作。

优点

  • 高内聚低耦合，每个组件可独立测试
  • 扩展性好，可替换任意组件
  • 决策过程完全透明

缺点

  • 文件更多（~8 个核心文件）
  • 需要设计接口
C

事件驱动管道 — 拦截器链 + 事件总线

每个决策环节是管道中的一个处理器，通过事件总线通信。用户可以任意插入/移除中间件。

优点

  • 最大灵活性，步骤可重排
  • 插件化，第三方可贡献中间件

缺点

  • 过度设计，v1 不需要
  • 事件驱动调试困难
  • 性能开销
